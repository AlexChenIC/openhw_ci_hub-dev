# Adding a New Repository to CI Hub

This guide explains how to onboard a new OpenHW repository so that its CI results appear in the CI Hub Dashboard and it can use the shared Tier 1/2 workflows.

---

## Prerequisites

The target repository must:
1. Use a RISC-V toolchain available on GitHub Actions standard runners (or specify a self-hosted runner)
2. Have a simulation entry point that can be wrapped in a `bash` or `python3` command
3. Have at least one test to run for Tier 1 (smoke test)

---

## Step 1: Add an entry to `config/repos.yml`

```yaml
- owner: "my-org"
  repo:  "my-riscv-core"
  display_name: "My RISC-V Core"
  description: "Short description for the dashboard"
  workflows:
    riscv-tier1: "tier1.yml"    # filename of the thin wrapper in the target repo
    riscv-tier2: "tier2.yml"
  matrix_config: "config/my-riscv-core-matrix.yml"
  active: true
```

---

## Step 2: Create a matrix config file

Create `config/my-riscv-core-matrix.yml`:

```yaml
# Known configuration names (used by parser.py for job name matching)
known_configs:
  - my_config_rv32
  - my_config_rv64

# Test suite display order in the dashboard matrix
known_suites:
  - smoke-test
  - arch-test
  - compliance
```

---

## Step 3: Create thin wrapper files in the target repo

### `tier1.yml` (PR Gate, ~15 min target)

```yaml
name: My Core Tier1 PR Gate
on:
  pull_request:
    branches: [master, main]
  workflow_dispatch:

concurrency:
  group: mycore-tier1-${{ github.ref }}
  cancel-in-progress: true

jobs:
  tier1-my-config:
    uses: my-org/ci-hub/.github/workflows/riscv-tier1.yml@main
    with:
      arch:     "RV32"           # RV32 or RV64
      target:   "my_config_rv32"
      testcase: "smoke-test"
      mode:     "script"         # script | testlist | custom_c_test
    secrets: inherit
```

### Job naming convention (critical for dashboard)

The job `name` generated by the reusable workflow must follow this format:

```
"{arch} Tier{N} {config} / {testcase}"
```

Examples:
- `"RV32 Tier1 cv32a65x / smoke-tests-cv32a65x"`
- `"RV64 Tier2 cv64a6_imafdc_sv39_hpdcache_wb / dv-riscv-arch-test"`

This is set automatically by `riscv-tier1.yml` / `riscv-tier2.yml` from the `arch`, `target`, and `testcase` inputs.

**For non-CVA6 repos with a different prefix** (e.g. `CV32E20`), extend the regex in `parser.py`:

```python
# In parser.py, line ~44
_RE_TIER = re.compile(
    r"^(RV(?:32|64)|CV\d+\w*)\s+Tier[12]\s+(.+?)\s*/\s*(.+)$"
)
```

The current regex already handles `CV32E20`; add new prefixes here if needed.

## Step 4: Handle simulation tool differences

The `setup-riscv-env` action installs Verilator + Spike + the RISC-V toolchain by default. If your repo uses different tools, override the inputs:

```yaml
# In the reusable workflow call inside riscv-tier1.yml:
jobs:
  setup:
    steps:
      - uses: my-org/ci-hub/.github/actions/setup-riscv-env@main
        with:
          install-tools:            'true'
          setup-script:             'my-repo/ci/setup-my-tools.sh'
          toolchain-install-script: 'my-repo/ci/install-toolchain.sh'
          # Disable Verilator/Spike caching if not used:
          verilator-install-script: '/dev/null'
          spike-install-script:     '/dev/null'
          spike-submodule-path:     ''
```

---

## Step 5: Test the integration

1. Open a test PR in the target repo → Tier 1 should appear as a PR status check
2. Manually run `collect-results.yml` in ci-hub
3. Manually run `deploy-dashboard.yml` in ci-hub
4. Check that the new repo's data appears in the dashboard

---

## Step 6: Enable the nightly trigger

Add a new job to `trigger-nightly.yml`:

```yaml
trigger-my-core-tier2:
  runs-on: ubuntu-latest
  steps:
    - run: |
        gh workflow run tier2.yml \
          --repo "my-org/my-riscv-core" \
          --ref  "master"
      env:
        GH_TOKEN: ${{ secrets.DISPATCH_TOKEN }}
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Jobs not appearing in dashboard matrix | Job name format wrong | Check `arch`, `target`, `testcase` inputs match the expected format |
| `parse_job_name` returns `None` | Unknown arch prefix or config | Add to `parser.py` `_RE_TIER` regex or `_DEFAULT_KNOWN_CONFIGS` |
| Dashboard shows "No data" | Data not collected yet | Run `collect-results.yml` manually |
| `gh api` fails in `collect-results.yml` | Wrong workflow filename in `repos.yml` | Check `workflows.riscv-tier1` matches the actual `.yml` filename |
| Nightly trigger fails | `DISPATCH_TOKEN` missing or expired | Regenerate PAT with `workflow` scope, update secret |
